# Atlas WebServer 面试答辩攻防手册

这份文档专门准备“面试官质疑项目”的场景。它不是普通问答题库，而是教你如何把压力问题回答成项目亮点，同时不夸大边界。

## 1. 总体答辩策略

Atlas 面试最容易被追问的点不是“接口有哪些”，而是：

- 你是不是真的理解 WebServer 底层？
- 网盘业务是不是只是 CRUD？
- 文件和 DB 一致性怎么保证？
- 本地磁盘怎么多机部署？
- 为什么限流组件要拆出去？
- 这个项目到底是不是生产级？
- 你能不能承认不足，并给出合理演进方案？

回答时建议采用四段式：

```text
1. 先给结论
2. 解释当前实现
3. 承认边界
4. 给生产化方案
```

不要一上来堆术语，也不要说“都实现了”。面试官更看重你是否知道当前方案解决了什么、没有解决什么、下一步怎么补。

## 2. 项目定位被质疑

### 问：你这个是不是普通 WebServer demo 套了几个接口？

答：

```text
不是。普通 WebServer demo 通常只做到静态文件、echo 或简单 HTTP 解析。Atlas 的底层保留了 epoll、非阻塞 socket、主从 Reactor、线程池、HTTP parser、chunked 和 multipart；业务层继续做了账号会话、文件上传下载、目录、回收站、分享、操作审计、SHA-256 去重、配额和限流。它的重点不是接口数量，而是把网络层、协议层、业务事务和文件一致性连成一个完整后端系统。
```

可以继续展开：

- 底层网络：`app/webserver.cpp`、`app/webserver_sub_reactor.cpp`
- 连接生命周期：`http/core/connection.cpp`
- 文件上传：`service/files/upload_service.cpp`
- MySQL 持久化：`repo/mysql/file_repository.cpp`

### 问：这个项目是不是生产级网盘？

答：

```text
我不会把它说成完整生产级网盘。它是一个功能闭环和工程验证比较完整的 C++ 后端项目，能展示网络编程、HTTP 协议、MySQL 事务、文件一致性、限流和测试体系。但生产级网盘还需要对象存储、多实例无状态部署、后台异步清理、分片上传、断点续传、Range 下载、文件预览、权限协作和更完整的可观测性。
```

这类问题要主动承认边界。承认边界不是扣分，能讲出升级路线才是加分。

## 3. WebServer 底层追问

### 问：你说用了 Reactor，具体怎么体现？

答：

```text
服务端主线程通过 epoll 监听 listenfd，负责 accept 新连接；连接建立后分发给 SubReactor。每个 SubReactor 有自己的 epoll，监听连接上的读写事件。读事件到来后先把 socket 数据读到连接缓冲区，再把请求处理投递到线程池。工作线程完成 HTTP 解析、路由、业务和响应构造后，连接切换到写事件，由 SubReactor 负责发送响应。
```

关键点：

- 主 Reactor 不处理所有业务，否则 accept 会被慢请求拖住。
- SubReactor 管连接事件。
- 线程池处理业务逻辑和数据库访问。
- `EPOLLONESHOT` 防止同一个连接被多个线程并发处理。

### 问：为什么不用每个连接一个线程？

答：

```text
每连接一线程在连接数上来后会有大量线程栈、上下文切换和调度成本，而且慢连接会长期占住线程。epoll 模型用少量线程监听大量 fd，只有事件就绪时才处理；线程池只承接真正需要 CPU 或数据库访问的任务，更适合高并发连接场景。
```

### 问：ET 模式为什么必须非阻塞？

答：

```text
ET 是边沿触发，内核只在状态变化时通知一次。如果读事件来了没有一次读到 EAGAIN，缓冲区里剩余数据可能不会再次触发通知。socket 必须非阻塞，否则循环读到没有数据时会卡死工作线程或事件线程。
```

## 4. HTTP 协议追问

### 问：HTTP 半包和粘包怎么处理？

答：

```text
TCP 是字节流，不保留 HTTP 请求边界。Atlas 的连接对象维护读缓冲区，每次 read_once 只是把当前可读字节追加进去，HTTP parser 根据 header、Content-Length、chunked 或 multipart 状态判断请求是否完整。请求没完整就继续等下一次读事件，而不是假设一次 read 就是一个完整请求。
```

### 问：multipart 上传为什么复杂？

答：

```text
multipart 不是普通 JSON，它包含 boundary、多个 part、文件内容和普通字段。上传不能把整个大文件一次性放内存里，所以要边解析边写临时文件，同时处理 boundary 跨 buffer、文件名校验、大小限制、失败清理和最终事务提交。
```

### 问：为什么要先写临时文件？

答：

```text
因为上传在业务校验前就已经开始接收字节流。如果直接写正式位置，请求失败会留下脏文件。先写临时文件可以在解析失败、鉴权失败、配额不足、DB 失败时统一清理；只有业务事务成功后，才把临时文件转为正式物理文件或因为去重而删除。
```

## 5. 数据库和文件一致性追问

### 问：files 和 physical_files 为什么拆开？

答：

```text
files 是用户视角的逻辑文件，记录文件名、目录、删除状态、公开状态等；physical_files 是真实物理内容，记录 sha256、stored_name、size 和 ref_count。同一个物理内容可以被多个用户或多个逻辑文件引用，这样才能支持 SHA-256 去重、秒传扩展和物理文件引用计数。
```

### 问：ref_count 为什么用 trigger 维护？

答：

```text
ref_count 是跨业务路径的派生数据，上传、删除、恢复、永久删除都可能影响它。如果全部放在应用层手动维护，很容易某个路径漏更新。trigger 把引用计数维护收敛到数据库层，只要 files 记录变化，就能统一维护 physical_files.ref_count。
```

### 问：DB 和磁盘能不能完全原子？

答：

```text
不能。MySQL 事务只能覆盖数据库，不能覆盖本地磁盘 rename/delete，也不能覆盖对象存储 put/delete。当前项目通过顺序约束、临时文件和一致性检查降低风险；生产化要引入对象状态字段、outbox 后台任务和巡检补偿，把不可原子的外部操作变成可重试、可观测的最终一致流程。
```

### 问：为什么永久删除要先 commit DB 再删磁盘？

答：

```text
因为 DB 是用户可见真相。如果先删磁盘再提交 DB，DB 回滚后用户还会看到文件，但物理文件已经没了，这是用户可见的数据损坏。先提交 DB 再删磁盘，最坏情况是留下孤儿物理文件，它不影响用户视图，可以后台清理。
```

## 6. 并发上传追问

### 问：并发上传同一个文件，怎么避免生成多份物理文件？

答：

```text
核心靠 sha256 唯一约束和事务。上传先计算内容 hash，进入事务后查询 physical_files 是否已有相同 sha256。如果没有就尝试插入物理记录；并发情况下唯一约束会保证同一个 hash 只能成功插入一条，其他事务发现冲突后复用已有 physical_id。这样逻辑文件可以有多条，但物理对象只有一份。
```

### 问：并发上传怎么防止用户配额被打穿？

答：

```text
配额检查必须放在服务端事务内，不能只靠前端。并发上传时要锁住用户相关行或使用等价的序列化手段，先计算当前已用容量和本次新增容量，确认不超过 quota 后再插入文件记录。否则两个请求可能同时看到“还有空间”，最后总量超出限制。
```

### 问：上传失败会不会留下临时文件？

答：

```text
正常失败路径会清理临时文件；同时项目有专项测试覆盖上传失败清理。生产化还需要后台 tmp 清理任务，定期删除超过 TTL 的临时文件，防止进程崩溃或机器重启导致临时文件残留。
```

## 7. 限流解耦追问

### 问：Atlas 和 Redis-Limiter 是什么关系？

答：

```text
Atlas 是网盘后端业务系统，Redis-Limiter 是可复用的分布式限流组件。Atlas 不复制限流组件源码，只在登录和注册入口构造业务 key，然后调用限流适配层。这样两个项目可以分别写在简历上：Atlas 展示 C++ WebServer 和网盘业务一致性，Redis-Limiter 展示 Redis Lua、令牌桶、滑动窗口、pybind11 和组件化设计。
```

### 问：为什么不把限流代码直接放进 Atlas？

答：

```text
因为限流是通用基础能力，不只网盘登录注册需要。拆成独立组件后，其他 C++ 或 Python 服务也能复用；Atlas 只依赖接口和业务 key 设计，不绑定具体算法实现。这体现的是组件边界，而不是为了凑项目数。
```

### 问：allow() 是否幂等？

答：

```text
不是。限流 allow 本质上会扣减令牌或写入窗口记录，重试一次就可能多消耗一次额度。业务调用时不能把已经发出的 allow 盲目重试。正确做法是明确调用边界，Redis 连接失败时进入 fallback；如果要支持业务级幂等，需要业务请求自己携带 request_id，并在业务层处理。
```

## 8. 安全追问

### 问：session token 怎么生成？

答：

```text
token 应该使用安全随机数生成，而不是 username、时间戳或简单 hash。生成后作为 Bearer Token 返回，服务端保存会话记录和过期时间。生产化还建议只保存 token hash，避免 DB 泄露后 token 被直接冒用。
```

### 问：密码为什么不能直接 SHA-256？

答：

```text
普通 SHA-256 太快，不适合存密码。攻击者拿到 hash 后可以高速暴力枚举。项目使用 PBKDF2-HMAC-SHA256 这类慢哈希思路，通过 salt 和迭代提高破解成本。生产上还可以使用 Argon2、bcrypt 或 scrypt。
```

### 问：公开分享有什么风险？

答：

```text
公开分享会绕过登录态，所以必须有不可猜测 token、可选访问码、过期时间、下载次数限制和审计。生产还要增加访问频率限制、防盗链、敏感文件扫描和异常下载风控。
```

## 9. 生产化追问

### 问：多实例怎么部署？

答：

```text
当前本地磁盘版本不适合多实例直接部署。生产化要让 Atlas 实例无状态：会话和元数据进 MySQL/Redis，文件对象进对象存储，本地只保留临时文件和日志。负载均衡可以把请求打到任意实例，实例通过共享存储访问同一份文件。
```

### 问：机器挂了怎么办？

答：

```text
当前单机版本机器挂了会影响服务和本地文件可用性，这是明确边界。生产方案会把文件持久化到对象存储，DB 做备份和高可用，Atlas 实例可以横向扩容；机器挂掉后流量切到其他实例，后台任务根据 outbox 继续补偿未完成操作。
```

### 问：下一步最值得做什么？

答：

```text
我会先做对象存储抽象和 outbox 补偿，因为这是从单机网盘走向多实例部署的核心边界。然后做 Range 下载和秒传 API，因为它们和现有下载、SHA-256 去重模型自然衔接。再往后做分片上传、后台任务、Prometheus 指标和权限协作。
```

## 10. 面试主动引导

如果面试官让你自由介绍，不要从“我有哪些接口”开始。建议这样组织：

```text
这个项目我会从三条主线讲。
第一条是 WebServer 底层：epoll、非阻塞 socket、主从 Reactor、线程池和 HTTP 解析。
第二条是网盘业务一致性：上传临时文件、SHA-256 去重、files/physical_files 拆表、ref_count、配额和 DB/磁盘补偿边界。
第三条是工程化：MySQL 迁移、限流组件解耦、smoke、并发一致性测试和 benchmark。
```

如果面试官偏底层，展开 Reactor。

如果面试官偏数据库，展开 files / physical_files / ref_count / quota。

如果面试官偏架构，展开本地磁盘边界、对象存储、outbox 和多实例。

如果面试官偏安全，展开密码、token、分享、限流和审计。

## 11. 不要这样回答

| 错误回答 | 问题 |
| --- | --- |
| “这是生产级网盘” | 容易被多实例、灾备、对象存储追问击穿 |
| “DB 和磁盘基本能保证一致” | 不能说基本，要明确不是原子，只能补偿 |
| “限流 allow 可以重试” | allow 会消耗额度，不是天然幂等 |
| “粘包半包靠 Content-Length 解决” | 只说一半，还要讲缓冲区和 parser 状态机 |
| “每个连接一个线程也可以” | 没体现 epoll/Reactor 的意义 |
| “文件名直接用用户上传的名字保存” | 有路径穿越和冲突风险 |

## 12. 最终 2 分钟答辩稿

```text
Atlas 是我用 C++17 做的 Linux 网盘后端。底层不是直接套框架，而是保留了 epoll、非阻塞 socket、主从 Reactor、线程池、HTTP/1.1 parser、chunked 和 multipart 解析；业务层实现了注册登录、Bearer Token、目录、上传下载、回收站、公开分享、操作审计、SHA-256 去重、配额控制和登录注册限流。

我觉得最值得讲的是上传和一致性。上传不是直接写正式文件，而是先写临时文件并计算 SHA-256，再在 MySQL 事务里检查目录、文件名、用户配额和 physical_files 去重。files 表表示用户视角的逻辑文件，physical_files 表表示真实物理内容，通过 ref_count 支持去重和物理文件清理。DB 和磁盘不能原子提交，所以当前通过顺序约束和一致性检查降低风险；生产化会进一步引入对象存储、outbox 后台任务和巡检补偿。

这个项目我不会说成完整生产级网盘，因为本地磁盘不适合多实例，断点续传、Range 下载、对象存储、权限协作和完整可观测性还需要继续补。它当前的定位是一个能完整演示 C++ 网络服务、HTTP 协议、数据库事务、文件一致性和工程验证能力的后端项目。
```
